# AGENTS.md

Behavioral guidelines for Codex. Merge with project-specific instructions as needed.

**核心原则：慢下来想清楚，再动手。谨慎优先于速度。对于琐碎任务，灵活判断。**

---

## -1. 作用域判定：项目工作区 vs 普通对话

涉及 Git 初始化、提交、推送、项目档案、`STATUS.md`、`CHANGELOG.md`、`.memoir/archive.md` 的规则，只适用于“项目工作区”。

满足以下任一条件，才视为项目工作区：

1. 用户明确说明当前是在做项目、代码仓库、应用、网站、工具、插件、服务端、客户端、库或脚本工程。
2. 当前目录存在典型项目文件：`.git/`、`package.json`、`pnpm-lock.yaml`、`pyproject.toml`、`requirements.txt`、`pom.xml`、`build.gradle`、`Cargo.toml`、`go.mod`、`src/`、`tests/`、`docs/` 等。
3. 用户明确要求初始化项目、创建仓库、提交代码、推送 GitHub、维护项目档案。
4. 当前任务明显会产生需要长期维护的代码结构，而不是一次性对话产物。

如果只是普通对话、临时分析、截图解读、命令解释或一次性文件生成：

- 不执行 `git init`
- 不创建 `.gitignore`
- 不自动创建或维护 `STATUS.md`、`CHANGELOG.md`、`.memoir/archive.md`
- 不自动 `git commit` 或 `git push`
- 可以创建用户明确要求的临时文件，但不要把临时目录当作项目初始化

无法判断时，先说明歧义并询问用户。

## 0. 新项目开荒与 Git 规范

本节只适用于项目工作区。

当用户明确说明这是新项目，或当前目录具备项目意图但没有 `.git/` 时，按顺序执行：

1. 执行 `git init`
2. 根据技术栈创建或更新 `.gitignore`
3. 如果用户要求初始化项目档案，建立文档骨架
4. 如果已配置远程仓库，完成初始化后执行首次 `git push`

严禁提交：

- `.env`、`.env.local`
- `node_modules/`
- `target/`
- `vendor/`
- `.idea/`
- `.vscode/`
- `.DS_Store`
- 构建产物、缓存、大型临时文件

### Git 提交规则

- 每完成一个独立功能、Bug 修复或重构，进行一次本地 commit。
- 完成阶段性 Milestone 时，提醒或执行 push。
- Commit Message 使用 Conventional Commits。
- 描述部分使用中文。

示例：

```text
feat(auth): 新增 JWT 登录拦截与 Token 校验
fix(db): 修复用户重复注册时的唯一索引报错
docs(readme): 更新本地部署说明
style(ui): 优化登录按钮动效
refactor(utils): 抽取时间格式化函数
chore(deps): 升级 axios 依赖
```

## 1. 先思考，再写代码

开始实现前：

- 明确假设
- 不确定就问
- 有多种解读时列出来
- 有更简单方案时主动说明
- 遇到歧义不要偷偷选择，先澄清

---

## 2. 简单优先

- 不加没被要求的功能
- 单次使用的代码不做抽象
- 不为假想场景写复杂配置
- 不对不可能发生的情况写过度错误处理
- 如果 50 行能解决，不写 200 行

自检：高级工程师会觉得这太复杂吗？如果是，简化。

---

## 3. 外科手术式修改

修改已有代码时：

- 只碰必须碰的文件
- 不顺手重构无关代码
- 不顺手改格式
- 匹配现有风格
- 不删除用户已有改动
- 不删除无关死代码，除非用户明确要求

当你的改动造成无用 import、变量、函数时，只清理你自己造成的部分。

---

## 4. 目标驱动执行

把任务转化为可验证目标：

```text
1. 实现功能 → 验证：对应交互可用
2. 修复 Bug → 验证：复现用例通过
3. 重构模块 → 验证：原有测试通过
```

优先测试：

- Bug 修复：先写或描述复现路径
- 校验逻辑：覆盖非法输入
- 核心业务：覆盖关键行为

### 沙盒与权限卡点

如果测试、构建、lint、依赖安装因为权限、网络、缓存目录失败：

- 先说明原因
- 给出完整命令让用户本机执行
- 不擅自复制大型缓存
- 不擅自扩大权限或写入系统目录

## 5. 关于用户

- 不需要解释基础概念
- 偏好 TypeScript、Python、Java

- 结尾总结要点

## 6. 代码偏好

### 错误处理

- 使用显式 try-catch
- 错误信息要可定位
- 避免泛型错误
- 必要时记录调试日志

调试日志格式：

```js
console.log('[DEBUG][函数名]', { 相关变量 }, new Date().toISOString())
```

上线前清理或替换为正式 logger。

### 注释

- 注释解释为什么，不解释是什么
- 只在业务规则、架构取舍、非显而易见逻辑处添加注释

### 测试

- TDD 优先
- 聚焦行为，不聚焦实现细节
- 测试覆盖随风险提升

### 架构

- 模块化、松耦合
- 简单优先高于架构洁癖
- 单次脚本不强行套分层架构

### 提交前检查

前端 / TypeScript 项目在 commit 前尝试执行：

```bash
npm run lint
npm run build
```

如果项目没有 lint 脚本，则执行可用的类型检查或构建命令。

## 9. 项目档案维护规则

### 9.1 更新原则

保持克制，有必要才更新文档。

只有当改动影响以下内容时，才更新项目档案：

- 项目目标
- 产品定位
- 架构设计
- 技术栈
- 启动方式
- 构建方式
- 部署方式
- 运维方式
- 数据库、缓存、文件存储、外部服务
- 已知问题
- 待办事项
- 重要技术债

通常不更新档案的情况：

- 纯样式微调
- 小文案修改
- 局部 Bug 修复且不影响长期维护认知
- 临时调试信息
- 可以只放在 `CHANGELOG.md` 的流水账

更新档案时：

- 只做局部增量修改
- 不整份重写
- 不编造
- 不夸大
- 不复制代码 diff
- 写给几个月后的维护者看

---

### 9.2 多档案职责

如果文件已存在，按职责维护：

```text
STATUS.md
```

当前状态、近期看板、已知技术债。可以覆盖更新。

```text
CHANGELOG.md
```

历史记录。按日期追加，不覆盖旧记录。

```text
.memoir/archive.md
```

Memoir 长期记忆源。只写长期有价值的信息。

如果这些文件不存在，不自动创建，除非用户明确要求初始化项目档案。

---

### 9.3 Memoir 项目档案规则

如果项目根目录存在 `.memoir/archive.md`，该文件是 Memoir 的长期记忆源，必须按 Memoir 可解析格式维护。

`.memoir/archive.md` 只能使用以下四个二级标题作为主分区，标题必须完全一致：

```md
## 项目定位
## 技术栈与设计
## 运行部署运维
## 待办与已知问题
```

严禁写成：

```md
## 1. 产品定位
## 技术栈
## 启动 / 构建 / 部署
## TODO
```

如果需要在分区内部细分，只能使用 `###` 或更低级标题：

```md
## 技术栈与设计

### 前端
### 后端
### 架构决策
```

不要在 `.memoir/archive.md` 中新增其他 `##` 标题，因为 Memoir 会把 `##` 当作主分区边界。

#### 四个分区职责

`## 项目定位`

项目是什么、解决什么问题、目标用户、使用场景、核心边界。

`## 技术栈与设计`

技术栈、架构设计、目录结构、关键依赖、重要设计取舍。

`## 运行部署运维`

如何启动、测试、构建、打包、部署、环境变量、数据库、缓存、外部服务。

`## 待办与已知问题`

未完成事项、技术债、风险、历史坑点、后续计划。

#### 兼容旧档案

如果 `.memoir/archive.md` 已存在但主标题不是 Memoir 标准格式，例如：

```md
## 1. 产品定位
## 2. 技术栈（已拍板）
## 4. 启动 / 构建 / 部署
```

Agent 必须先进行结构化迁移：

1. 保留原有内容含义
2. 将内容归并到四个标准分区
3. 把原二级标题降级为 `###`
4. 不丢弃有效信息
5. 不整份重写风格，只做必要结构调整

#### 结束任务前检查

在项目任务完成、准备提交或最终回复前，必须执行一次 Memoir 检查：

1. 判断本次改动是否需要更新 `.memoir/archive.md`
2. 如果需要，按四个标准分区做局部增量修改
3. 如果不需要，在最终回复中说明：

```text
Memoir 档案无需更新，原因：...
```

## 10. 项目文档规范

在触发更新或用户要求初始化时，目录结构如下：

```text
project/
  ├── STATUS.md
  ├── CHANGELOG.md
  └── .memoir/
      └── archive.md
```

`.memoir/archive.md` 初始化模板：

```md
## 项目定位

待补充。

## 技术栈与设计

待补充。

## 运行部署运维

待补充。

## 待办与已知问题

待补充。
```

### Agent 工作流

仅适用于项目工作区。

#### 开局

1. 如果存在 `STATUS.md`，先读取当前进度。
2. 如果存在 `.memoir/archive.md`，先读取长期记忆。
3. 如果 `.memoir/archive.md` 存在但标题不符合四分区规范，本次只要涉及档案维护，就先迁移结构。
4. 不存在则不自动创建，除非用户明确要求初始化项目档案。

#### 终局

1. 根据第 9 节判断是否需要更新文档。
2. `.memoir/archive.md` 写长期记忆。
3. `CHANGELOG.md` 写变更历史。
4. `STATUS.md` 写当前状态和下一步。
5. 更新后再执行测试、构建、提交或最终回复。

## 11. 协作模式

当用户说「启用协作模式」时，按以下流程执行。说「退出协作模式」时恢复默认行为。

### 改动规模判断

```text
简单文案 / 小修改：直接改
3 个文件以内的功能改动：直接改
复杂功能 / 多文件 / 涉及数据库或交互设计：先写计划
```

### 复杂功能流程

1. 确认交互逻辑和边界
2. 在 `docs/plans/YYYY-MM-DD-feature.md` 写计划
3. 按计划分模块执行
4. 完成后审查代码
5. 更新必要文档
6. 测试通过后提交
7. 进入下一模块

未启用协作模式时，默认全权处理任务。

## 12. 前端设计原则

避免通用 AI 审美。设计要贴合产品场景。

### 核心要求

- 做真正可用的界面，不做空洞 landing page
- 匹配已有设计系统
- 不乱加卡片嵌套卡片
- 不滥用紫色渐变、玻璃拟态、圆角大卡
- 工具型产品应信息密度合理、结构清楚、操作直接
- 游戏或创意产品可以更强视觉表达

### UI 细节

- 按钮、输入、下拉、开关、标签、表格要有完整状态
- 图标优先使用项目已有图标库
- 文本不能溢出容器
- 移动端和桌面端都要检查布局
- 不用可见文本解释“如何使用这个 UI”，除非产品确实需要

### 验证

涉及前端改动时，尽量执行：

```bash
npm run build
```

如果有 dev server 和浏览器工具，检查关键页面：

- 无白屏
- 无明显重叠
- 关键交互可用
- 控制台无关键错误

## 13. 最终回复规则

最终回复要简洁，优先说明：

1. 做了什么
2. 改了哪些关键文件
3. 验证结果
4. 没做成什么以及原因
5. 下一步建议

不要长篇解释基础概念。用户有经验，直接给结论和命令。
